# Frontend Audit (Phase M2)

## Текущая архитектура
- Проект состоит из статических HTML-страниц: `index`, `callback`, `dashboard`, `upload`, `transcription`, `quizzes`, `quiz`.
- Каждая страница содержит собственные inline `CSS` и `JS`.
- API-база (`CONFIG.API_URL`) дублируется в каждом файле.
- Единая визуальная тема есть, но без централизованных токенов/общих стилей.

## Карта экранов
- `index.html`: лендинг + вход через Яндекс OAuth.
- `callback.html`: обработка OAuth callback.
- `dashboard.html`: список транскрипций + базовая аналитика.
- `upload.html`: форма загрузки аудио.
- `transcription.html`: просмотр транскрипции + запуск генерации квиза.
- `quizzes.html`: список квизов.
- `quiz.html`: прохождение квиза.

## Основные UX/UI-проблемы
1. Повторяющийся код: сайдбар, mobile-nav, переменные темы и кнопки копируются между страницами.
2. Нет явной дизайн-системы: трудно быстро менять стиль во всём приложении.
3. Визуальная плотность неравномерна: карточки и отступы отличаются между экранами.
4. Много emoji-иконок в навигации и статусах, что снижает консистентность интерфейса.
5. Токены цветов и шрифтов разнесены по файлам, высок риск рассинхрона при редизайне.

## Приоритет редизайна
1. Общий каркас (навигация, хедер, типографика, цветовые токены).
2. `dashboard.html` как главный экран продукта.
3. `upload.html` и `transcription.html` как ключевой user flow.
4. `quizzes.html` и `quiz.html` как secondary flow.
5. `index.html` и `callback.html` как входной сценарий.

## Рекомендованная стратегия внедрения
1. Сначала согласовать 3 визуальных концепта в изолированных HTML-макетах.
2. После утверждения выделить единый слой токенов/базовых компонентов.
3. Переносить стиль по экранам при сохранении текущей JS-логики и API-контрактов.
